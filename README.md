# GBA Docker Template

Template repository for building Game Boy Advance homebrew projects with `devkitARM` in containers.

It provides three build images and a Makefile workflow that mounts `./source` into the container and runs your build command there.

## Images

- `base` (`dkarm_base:local`): `devkitARM` + core build tools
- `dusk` (`dkarm_dusk:local`): base image + extra utilities
- `butano` (`dkarm_butano:local`): dusk image + Butano engine

## Repository Layout

- `Makefile`: top-level container build/compile targets
- `docker/`: Dockerfiles for each image flavor
- `source/`: your GBA project (includes `source/Makefile` and `source/src/main.cpp`)

## Quick Start

1. Build the image you want:

```sh
make docker-build-butano
```

2. Build the ROM from inside the container:

```sh
make compile-butano CMD="make -j4"
```

3. Output:

- `source/source.gba` is generated by the project build
- top-level `./GBA_docker_template.gba` is copied automatically by `compile-butano`

## Common Commands

Build images:

```sh
make docker-build-base
make docker-build-dusk
make docker-build-butano
```

Run custom commands in containers:

```sh
make compile-base CMD="bash"
make compile-dusk CMD="make clean && make"
make compile-butano CMD="make -j4"
```

Clean Docker stamp cache:

```sh
make clean-docker-stamps
```

## Runtime Behavior

Before build/compile targets run, `check-container-runtime` verifies container runtime availability.

- Supports `docker` and `podman`
- Auto-starts Docker Desktop on:
  - macOS (via `open -a Docker`)
  - WSL2 Ubuntu (via `powershell.exe` / `cmd.exe`)
- Wait timeout can be adjusted:

```sh
make DOCKER_START_TIMEOUT=120 compile-base CMD="make"
```

You can force a runtime explicitly:

```sh
make CONTAINER_RUNTIME=podman compile-base CMD="make"
```

Compile targets also support `AUTO_CLEAN_MAKE`:

- `AUTO_CLEAN_MAKE=auto` (default): clean only when stale dependency paths are detected
- `AUTO_CLEAN_MAKE=1`: always run `make clean` before `make*` commands
- `AUTO_CLEAN_MAKE=0`: never auto-clean

## Debugging with mGBA (Host + Dev Container)

This repo includes VS Code tasks and launch configuration for GDB attach debugging with host mGBA auto-launch:

- Build task: `Build ROM (debug)` (`.vscode/tasks.json`)
- Host launcher task: `Launch mGBA on host (bridge)` (`.vscode/tasks.json`)
- Debug config: `Attach to mGBA GDB stub (host:2345)` (`.vscode/launch.json`)
- Host bridge script: `scripts/mgba_host_bridge.py`

Workflow:

1. Open the repo in the dev container.
2. Press `F5` and run the debug configuration:
   `Attach to mGBA GDB stub (host:2345)`.

Notes:

- The devcontainer `initializeCommand` attempts to auto-start the host bridge on macOS:
  `scripts/start_mgba_bridge.sh`.
- If auto-start fails, run this once on the host:

```sh
bash scripts/start_mgba_bridge.sh
```

- Auto-start attempts are logged here:
  `.devcontainer/mgba-bridge-init.log`
- If it still fails, check bridge logs on host:

```sh
cat /tmp/mgba-host-bridge.log
```

- Default mGBA binary path is `/Applications/mGBA.app/Contents/MacOS/mGBA`.
  Override with host env var `MGBA_BIN` if needed.
- The debugger connects to `host.docker.internal:2345` from the container.
- Symbols are loaded from `source/source.elf`.
